K8s Learning Lab

Five hands-on tasks to learn Kubernetes — run them in your terminal above

Your environment is ready.

Open your terminal at learn-[name].bp31app.com and work through the tasks below. Replace [name] with your username wherever you see it. Everything you do is scoped to your own namespace — you can't break anything outside it.

1
Run your first Pod
Goal: understand Pods — the smallest deployable unit in Kubernetes

A Pod is one or more containers running together. Start a simple nginx container.

# Start a one-shot pod
kubectl run hello --image=nginx --restart=Never

# Watch it start up
kubectl get pods

# Read its logs
kubectl logs hello

# Inspect the pod in detail
kubectl describe pod hello
What you learned: Pods are ephemeral. If it crashes, it's gone. That's why real workloads use Deployments (Task 2).
2
Create a Deployment
Goal: understand Deployments — self-healing, declarative pod management

A Deployment tells Kubernetes "I always want 3 replicas of this app". K8s will keep that true — even if pods crash.

# Create a deployment with 3 replicas
kubectl create deployment myapp --image=nginx --replicas=3

# Check the deployment
kubectl get deployments

# See the pods it manages
kubectl get pods -l app=myapp

# Delete a pod and watch it get recreated
kubectl delete pod <one-of-the-pod-names>
kubectl get pods -l app=myapp
What you learned: Deployments manage a ReplicaSet which manages Pods. Deleting a Pod doesn't kill the app — the Deployment respawns it.
3
Expose with a Service
Goal: understand Services — stable network endpoints for pods

Pods get random IPs that change on restart. A Service gives you a stable name and load-balances across all pods.

# Create a ClusterIP service for myapp
kubectl expose deployment myapp --port=80 --type=ClusterIP

# See the service and its cluster IP
kubectl get svc

# Hit the service from inside the cluster
kubectl run test --image=busybox --rm -it --restart=Never \
  -- wget -qO- http://myapp

# Kubernetes also provides DNS — same request via name
kubectl run test --image=busybox --rm -it --restart=Never \
  -- wget -qO- http://myapp.learn-[name].svc.cluster.local
What you learned: Services provide stable DNS names and load balance across all matching pods. ClusterIP = cluster-internal only.
4
Deploy to the internet
Goal: understand Ingress — routing external HTTP traffic into your cluster

An Ingress connects a public hostname to a Service. Your subdomain learn-[name]-app.bp31app.com is already wired up — just create the Ingress rule. 

# Create an Ingress to expose myapp to the internet
kubectl create ingress myapp \
  --rule="learn-[name]-app.bp31app.com/*=myapp:80"

# Check the ingress was created
kubectl get ingress

# Wait ~30 seconds, then visit your URL:
# https://learn-[name]-app.bp31app.com
What you learned: Traefik (the ingress controller) watches for Ingress resources and automatically routes traffic. Real traffic, real internet, your deployment.
Replace [name] with your actual username in both the command and the URL.
5
Scale and observe
Goal: understand scaling, rolling updates, and self-healing

Kubernetes makes horizontal scaling trivial. Watch it happen in real-time.

# Scale up to 5 replicas
kubectl scale deployment myapp --replicas=5

# Watch pods spin up in real-time (Ctrl+C to stop)
kubectl get pods -w

# Check rollout status
kubectl rollout status deployment/myapp

# Scale back down
kubectl scale deployment myapp --replicas=1

# Update the image (triggers a rolling update)
kubectl set image deployment/myapp nginx=nginx:alpine

# Watch the rolling update
kubectl rollout status deployment/myapp

# See rollout history
kubectl rollout history deployment/myapp
What you learned: Rolling updates replace pods one by one with zero downtime. kubectl rollout undo rolls back if something goes wrong.
